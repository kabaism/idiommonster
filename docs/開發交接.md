# 成語怪物｜開發交接

最後更新：2026-04-11

## 1. 專案目前定位

- 這版是「純前端靜態頁」遊戲（HTML/CSS/JS），無後端。
- 核心流程已可跑：首頁選模式/難度 -> 題目作答 -> 結算 -> 錯題回顧。
- 題庫已拆獨立檔，方便後續做管理介面或接資料來源。

## 2. 題庫現況（已確認）

- `basic`：50 題
- `advanced`：50 題
- `master`：50 題

總計 150 題。

## 3. 主要檔案地圖

- `index.html`：畫面骨架、各 screen 結構、boot debug 面板、script 載入順序
- `styles.css`：整體視覺、版面、動畫、RWD
- `script.js`：遊戲主流程、鍵盤/滑鼠事件、計時與判題、結算與錯題回顧
- `idiom-bank.js`：題庫資料 + 正規化（`window.IDIOM_BANK`）
- `game-config.js`：回合數、秒數、功能開關（搞笑選項、圖片顯示）
- `docs/題庫資料格式.md`：題目欄位說明
- `docs/部署與跨機開發.md`：GitHub Pages 與跨機開發流程

## 4. 目前可用功能

### 首頁
- 模式：單人/雙人
- 難度：基礎/進階/高級
- 快捷鍵：`S`、`D` 切模式；`1` `2` `3` 切難度；`Enter` 開始

### 遊戲中
- 8 題一局（可在 config 調整）
- 10 秒倒數（可在 config 調整）
- 雙人熱鍵：Player1 `A/S/D/F`，Player2 `J/K/L/;`
- 雙人兩側選項順序獨立亂數
- 雙人選擇隱藏：只顯示「尚未選擇 / 已選擇，可更換 / 答案確認中...」
- 題目上方顯示格式：`第 X 題（共 Y 題）`

### 結算與回顧
- 結算畫面顯示分數與答對數
- 錯題回顧可看：題目、提示、你的作答、正解、該題選項
- 錯誤率排行榜（同局前幾名錯題）

## 5. 題庫資料設計（重要）

目前每題可用欄位：

```js
{
  idiom: "亡羊補牢",
  meaning: "出了問題後立刻補救，仍然來得及",
  hint: "重點是補救，不是放棄",
  wrongMeanings: [
    "錯誤選項1",
    "錯誤選項2",
    "錯誤選項3"
  ],
  funnyMeaning: "搞笑選項",
  image: "",
  imageAlt: "亡羊補牢 插圖"
}
```

補充：
- `wrongMeanings` 沒填時，系統會從同難度池自動產生干擾選項。
- `funnyMeaning` 目前已為「每題資料驅動」，建議維持一題一個專屬內容。
- `showIdiomImage = false` 時圖片欄位不顯示，但資料可先保留。

## 6. 偵錯方式（第一優先）

### boot debug 模式
- 在網址後加 `?debug=1`
- 左下角會顯示 script 載入與錯誤訊息

例如：
- `[load] idiom-bank.js ok`
- `[load] game-config.js ok`
- `[error] ...`

### 常見故障排查

1. 無法點選/快捷鍵無效  
先看 console 是否有語法錯（script 中斷後事件不會綁定）。

2. 題庫載入失敗  
確認 `index.html` 內 script 順序：`idiom-bank.js` -> `game-config.js` -> `script.js`。

3. 題目選項品質問題（語意太近）  
優先在該題填 `wrongMeanings`，不要只依賴自動混搭。

## 7. 已知待修清單（建議照順序）

1. 單人結算畫面仍需再驗證「只顯示單人資訊」版型（避免殘留雙人欄位視覺）。
2. 全題 `funnyMeaning` 建議改成真正逐題客製文案（目前有一部分會走預設生成邏輯）。
3. 針對「易混淆題」改為嚴格題本制：每題固定 `meaning + wrongMeanings + funnyMeaning`，降低雙正解風險。
4. 補一輪迴歸測試：
   - 首頁熱鍵與滑鼠可開始遊戲
   - 單人/雙人都能完整跑完一局
   - 錯題回顧顯示正確
   - RWD（手機寬度）排版不重疊

## 8. 部署與分享

- 目前最適合用 GitHub Pages（靜態站點，免後端）
- 詳細步驟：`docs/部署與跨機開發.md`

若未來需要後端（帳號、排行、雲端題庫）：
- 可選 Firebase / Supabase / 自架 API
- 建議先把 `idiom-bank` 結構穩定，再接 API，避免前後端同時大改

## 9. 跨機接續開發（Codex）

重點：
- 檔案會跟著 git 走，但「對話上下文」不會完整自動繼承。
- 所以每次交接請更新這份文件，並在 commit 訊息寫清楚改動與未完成事項。

推薦流程：
1. push 最新 commit 到 GitHub
2. 新機器 clone/pull
3. 開 Codex 後先讀：
   - `README.md`
   - `docs/開發交接.md`
   - `docs/題庫資料格式.md`
4. 再開始下一輪開發
